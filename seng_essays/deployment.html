What constrains how frequently an organisation can release new software?

1. Lack of automation in production pipelines; time is wasted when humans do repetitive, structured tasks slowly and inconsistently when they could be done quickly and reliably by machines.[1] For example, by implementing continuous delivery Paddy Power could provision automated test environments in minutes, whereas before it would take multiple teams weeks to find machines, install OS’s, and deploy and configure applications for testing.[2]

2. Lack of prompt, precise, and accessible feedback about code changes, both from automated tests and users, which helps to identify and resolve problems quickly. When issues arise (failing tests, performance drops, feature criticisms...), developers should know ASAP what happened in which build running in which environment and what source code changes were in that version.[3] [4] Without this feedback, problems might not be found until so many other changes have been added that the original cause is both harder to identify and harder to fix, therefore delaying a stable release. Having moved to CD, Paddy Power saved developers’ time by dropping unimportant features based on user feedback and reduced the number of open bugs by >90% by testing the whole codebase immediately whenever a change was made.2 

3. The need to minimise risk and comply with regulations[5]. Stricter tests and review procedures may be required if changes can disrupt critical operations or violate laws. One way investment banks mitigate the risk of releasing new software is by only updating their algorithmic trading systems on weekends, while stock exchanges are closed, which is safer but limits release frequency.

4. Working with independent software development contractors; it can be hard to resolve intellectual property conflicts (who owns what code) if many people make frequent small changes to the source, so in-house teams and contractors are forced to work in a highly sequential manner, wasting time taking distinct ‘turns’ to make changes.[6] Additionally, code developed in isolation from a larger team, as it often is in these cases, can take longer to integrate.
