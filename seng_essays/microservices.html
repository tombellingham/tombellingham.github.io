When is moving to a microservice architecture a good idea?

A microservice architecture consists of autonomous components which communicate with each other via web requests or remote procedure calls[1] and which are replaceable, independently deployable[2], and relatively small and simple compared to monoliths.[3]

Some of the main strengths of this architecture include:

  • Granular scalability: instances of a microservice can be deployed or destroyed dynamically3 to match available resources with unique service requirements[4] (up-time, response time, throughput, etc...).
  • Independent deployment: a microservice can be updated and deployed without rebuilding and rebooting the whole application1, so they are easy to maintain3 and ideal for teams that release small changes frequently.
  • Technological heterogeneity1: each microservice can use different languages, frameworks, and databases, etc... so teams can choose the best tools and configurations for each component and easily experiment with new technologies[5] - the only constraint is the communication technology itself3.

Some of the main drawbacks are:

  • Increased security risk; distributed microservices implemented with different technologies form a large surface attack area with no common security infrastructure, application internals are exposed through network APIs, and the complexity of interactions between components makes detecting and defusing threats harder.3
  • Remote calls between components are slower1 and less reliable than in-memory calls because they are constrained by latency, bandwidth and network glitches - there are more ways for an application to break if connections between components can fail.
  • It can be difficult to guarantee data consistency1 across microservices; there is no clear way to separate microservice domains.[6] 

Deciding whether to move from a monolith to a microservice model depends on evaluating the relative significance of these tradeoffs. In summary, though generic architectural advice must be tentative, it would probably be beneficial to make the move if the security and data consistency challenges are outweighed by the benefit of fine-grained scalability, fast releases, and technological heterogeneity.
