What benefits/problems come from deploying services as serverless functions?

‘Serverless’ (a.k.a Function-as-a-Service or FaaS) is a “programming model and architecture”[1] where a cloud platform provider executes code snippets in a scalable way on behalf of a developer[2] - the functions uploaded by developers are usually stateless1 event handlers[3], which are even more lightweight and fine-grained than virtual machines or containers1.

FaaS providers abstract away operational concerns1 and infrastructural plumbing3 so developers can focus on business logic1,3 and product features[4] and minimise the need for boilerplate code3. These providers have taken a “big step forward”2 for cloud computing with pay-as-you-go billing - only charging for actual execution time and data transfer - which can cut operational costs hugely since all idle time is free1,2,3. They also provide “seemingly unlimited”2 automatic scaling, so developers don’t need to worry about unpredictable high traffic or reserving resources that wouldn’t otherwise be needed3, thereby cutting down on capital expenditure for dedicated servers3. Serverless application development can be fast[5] because each function can be changed and deployed independently and the intrinsic modularity of event-based handlers lends well to parallelising work within or between teams3.

However, gains made by abstracting away operation are lost in restricted control1,4. Most FaaS providers don’t offer strong service-level guarantees1,2,3, but some developers need to understand how a cloud platform behaves under different loads1 - this opaqueness makes these platforms unsuitable for critical systems3. Some FaaS providers also put hard limits on execution time2 and RAM for individual invocations (900s and ≈3GB for AWS Lambda[6]); “serverless functions have limited expressiveness as they are built to scale”1; and some languages and libraries may not be supported1. There is also a risk of vendor lock-in because FaaS providers often only support their own ecosystem of cloud services1,3.

Serverless functions face performance drawbacks that can “fatally restrict”2 data processing efficiency; data must be shipped to the machine executing the function which is a “recurring anti-pattern among system designers”2 because network speed is “orders of magnitude slower than a single modern SSD”2 and bandwidth is split between every function running on a machine2 - this is particularly bad for ML models which require lots of training data2. Function invocations can have a ‘cold start’ latency on freshly provisioned instances1,3, inter-function messaging is slow because function instances aren’t network addressable so must communicate via an intermediary service2, and most provisioned VM’s are “uniform and fairly mundane”2 with no support for GPU acceleration or specialised hardware APIs2.

Serverless functions are ideal for bursty, compute intensive workloads1 of simple independent tasks2 that can each be completed quickly3, but they are not ideal for services that require low-latency and high-availability3 - whether fine-grained distributed applications2 or critical systems3.
